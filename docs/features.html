Key Features

Aster extends RocksDB with RocksGraph, a graph storage and traversal layer designed for workloads that combine frequent edge and vertex mutations with neighbor-lookup-heavy queries. The design is based on the Poly-LSM architecture described in the SIGMOD 2025 paper.

Poly-LSM Architecture

The core innovation in Aster is an adaptive edge update strategy that chooses the best write path for each vertex based on current workload characteristics.

Eager

Reads the existing adjacency list, merges the new edge in-place, and writes back via Put. Best for low-degree vertices with frequent lookups.

Lazy

Uses merge operators (AdjacentListMergeOp) for incremental writes, avoiding full adjacency reads. Best for high-degree vertices under heavy mutation.

Adaptive

Dynamically switches between eager and lazy at runtime, using degree estimates and update/lookup ratios to pick the optimal strategy per vertex.

In adaptive mode, Aster evaluates a cost model for each edge operation. The decision boundary depends on the vertex degree, block size, and the current read-to-write ratio — the same formula shown in the interactive demo on the landing page.

Adjacency Encoding

Aster supports two adjacency list storage formats, controlled by the encoding_type parameter at graph creation time.

Raw (ENCODING_TYPE_NONE)

Adjacency lists stored without compression. Provides the fastest read and write access at the cost of larger storage footprint.

Elias–Fano (ENCODING_TYPE_EFP)

Partitioned Elias–Fano compression. Significantly reduces storage size while still enabling efficient random access to individual neighbors.

Degree Approximation

When an exact degree count is not required, Aster provides O(1) estimation via GetDegreeApproximate(), avoiding the cost of reading and decoding the full adjacency list.

Morris Counter

Space-efficient logarithmic counter using FILTER_TYPE_MORRIS. Default estimator. Extremely compact representation.

The adaptive update policy uses these degree estimates internally to decide between eager and lazy writes — no extra configuration needed.

Property Support

Both vertices and edges can carry arbitrary key-value properties. Properties are stored in dedicated column families (vprop_val and eprop_val), separate from the adjacency data.

Capabilities

  • Write and read properties per vertex or per edge
  • Scan-based reverse lookup: find all vertices or edges matching a given property
  • Multiple properties per entity, stored as concatenated name-value pairs
The property lookup helpers (GetVerticesWithProperty, GetEdgesWithProperty) are convenience scan functions. They are useful for testing and small workloads, but they are not secondary-index lookups.

Built on RocksDB

Aster inherits the full production-grade infrastructure of RocksDB, the LSM-tree storage engine used in production at Facebook, ByteDance, LinkedIn, and many more.

What you get for free

  • Bloom filters for fast negative lookups
  • Direct I/O support for bypassing OS page cache
  • Pluggable compression codecs (zlib, lz4, snappy, zstd)
  • Column families for logical data separation
  • Write-ahead log and crash recovery
  • Extensive statistics and diagnostics via GetRocksDBStats()

Performance Highlights

The Poly-LSM design delivers substantial improvements over both standard LSM approaches and competing graph databases.

Highly scalable

Achieves significant comparative performance at billion-scale graphs where other approaches degrade.

Mixed workload robustness

Adaptive policy avoids worst-case degradation regardless of the read/write ratio.

Disk-native design

Scales to graphs that exceed available memory, backed by RocksDB’s proven LSM-tree storage.
