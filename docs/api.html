API & Examples

Aster’s public graph interface is exposed in include/rocksdb/graph.h. The main abstraction is RocksGraph, which wraps RocksDB with graph-aware storage, edge update policies, adjacency encoding, degree queries, and property support.

Each vertex is stored as one record that contains both its outgoing and incoming neighbors. Vertex properties and edge properties are stored separately.

Where the API lives

Main entry points

  • include/rocksdb/graph.h — public APIs (data structures, encoding helpers, metadata I/O)
  • db/graph.cc — core implementation of storage logic and graph operations
  • graph_test/graph_example.cc — smallest end-to-end example

Data Model & Storage Layout

Identifiers

  • node_id_t — 64-bit integer vertex identifier
  • edge_id_t — 64-bit edge identifier type used internally

Storage context

  • Each vertex record stores both outgoing and incoming neighbors
  • Vertex properties live in vprop_val
  • Edge properties live in eprop_val
  • Graph metadata stores vertex and edge counts (n, m)

Returned structures

  • Edges is the in-memory structure used to represent one vertex’s outgoing and incoming neighbors. Stored adjacency is decoded into this structure when read.
  • Property stores a name and a value. Vertex properties are stored in vprop_val, and edge properties are stored in eprop_val.
GetAllEdges(...) returns adjacency through an Edges structure in memory. After you finish using it, call free_edges(...) to release the returned arrays.
In the public API, edges are usually identified by their (from, to) endpoints. The graph API works with directed edges. If you want undirected behavior, add both directions.

Example1 - Open a graph

A typical setup creates a RocksGraph instance with RocksDB options, an edge update policy, an encoding type, and a database path. options.create_if_missing = true tells RocksDB to create the database if it does not already exist at "/tmp/aster-demo". If a database is already stored at that path, RocksDB simply opens the existing one. 

#include "rocksdb/graph.h"
#include "rocksdb/options.h"

rocksdb::Options options;
options.create_if_missing = true;

rocksdb::RocksGraph graph(
    options,
    EDGE_UPDATE_ADAPTIVE,
    ENCODING_TYPE_NONE,
    false,  // auto_reinitialize
    "/tmp/aster-demo"
);
EDGE_UPDATE_ADAPTIVE means Aster does not always use just one update method. It decides at runtime whether to apply updates eagerly or lazily, depending on the current graph and workload characteristics.
"encoding_type" controls how adjacency lists are stored in the main graph storage. ENCODING_TYPE_NONE keeps raw adjacency lists, while ENCODING_TYPE_EFP stores them in a compressed Elias–Fano-style format.
Setting auto_reinitialize to true deletes the existing database at that path before opening it. For normal usage, keep it false unless you intentionally want a fresh DB.

Core graph APIs

Structure and mutation

  • AddVertex(node_id_t id)
  • AddEdge(node_id_t from, node_id_t to)
  • DeleteEdge(node_id_t from, node_id_t to)

Read and count

  • GetAllEdges(node_id_t src, Edges* edges)
  • GetOutDegree(node_id_t id)
  • GetInDegree(node_id_t id)
  • CountVertex() — get the number of vertices from the graph metadata
  • CountEdge() — get the number of edges from the graph metadata

Bulk load

  • AddVertexWithEdges(node_id_t id, std::vector<node_id_t>& out_neighbors, std::vector<node_id_t>& in_neighbors) — writes one new vertex and its full adjacency list in one step. It assumes the vertex is new and only writes that vertex’s own adjacency.
  • AddEdges(node_id_t from, std::vector<node_id_t>& tos, std::vector<node_id_t>& froms) — helper that creates one vertex record from a full outgoing-neighbor list and incoming-neighbor list.
  • AddVertexForBulkLoad() — increments the vertex count graph metadata during bulk loading.

Example2 - Minimal graph

This mirrors the same basic flow used by the tiny example in graph_test: add vertices, add edges, read neighbors, then delete one edge. 

#include <iostream>

rocksdb::node_id_t u = 1, v = 2, w = 3;

graph.AddVertex(u);
graph.AddVertex(v);
graph.AddVertex(w);

graph.AddEdge(u, v);
graph.AddEdge(u, w);

rocksdb::Edges edges;
if (graph.GetAllEdges(u, &edges).ok()) {
  for (uint32_t i = 0; i < edges.num_edges_out; ++i) {
    std::cout << edges.nxts_out[i].nxt << "\n";
  }
  free_edges(&edges);
}

graph.DeleteEdge(u, v);
GetAllEdges(...) fills temporary in-memory arrays inside Edges. Call free_edges(&edges) when you are done with that result. This only frees the returned arrays in your program. It does not delete anything from the database.

Properties APIs

Aster supports both vertex properties and edge properties. It also provides helper functions for finding vertices or edges with a matching property.

Write and read properties

  • AddVertexProperty(node_id_t id, Property prop)
  • AddEdgeProperty(node_id_t from, node_id_t to, Property prop)
  • GetVertexProperty(node_id_t id, std::vector<Property>& props)
  • GetEdgeProperty(node_id_t from, node_id_t to, std::vector<Property>& props)

Property lookup helpers

  • GetVerticesWithProperty(Property prop) — returns a list of vertex IDs that have the given property.
  • GetEdgesWithProperty(Property prop) — returns a list of directed edges, represented as (from, to) pairs, that have the given property.
The lookup helpers are convenience scan helpers over the stored properties. They are useful for testing and small workloads, but they are not secondary-index lookups.

Property example

rocksdb::Property vp{.name = "kind", .value = "paper"};
graph.AddVertexProperty(u, vp);

std::vector<rocksdb::Property> vprops;
graph.GetVertexProperty(u, vprops);

rocksdb::Property ep{.name = "weight", .value = "1"};
graph.AddEdgeProperty(u, w, ep);

std::vector<rocksdb::Property> eprops;
graph.GetEdgeProperty(u, w, eprops);

auto matched = graph.GetVerticesWithProperty(vp);
AddVertexWithEdges(...) writes the adjacency for one vertex. It does not automatically update reverse edges on neighbor vertices for you.

Other APIs

Approximate, traversal, and diagnostics

  • GetDegreeApproximate(node_id_t id) — returns an approximate degree estimate for a vertex. It supports FILTER_TYPE_CMS (Count-Min Sketch) and FILTER_TYPE_MORRIS (Morris Counter), which are useful when a lightweight estimate is enough and an exact degree is not required.
  • SimpleWalk(node_id_t start, float decay_factor) — performs a random-walk style traversal starting from the given vertex. This API is mainly useful for traversal experiments or lightweight graph sampling.
  • GetRocksDBStats(std::string& stat) — retrieves internal RocksDB statistics as a string, which is useful for debugging, profiling, and understanding storage-level behavior.
  • printLSM(int column) — prints LSM-tree information for inspection, helping users examine how graph data is organized internally across storage levels.
